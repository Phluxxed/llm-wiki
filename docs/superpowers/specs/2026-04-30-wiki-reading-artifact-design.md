# Wiki Reading Artifact ŌĆö Design

**Date:** 2026-04-30
**Status:** Approved (pending implementation plan)
**Owner:** Vik Brummer

## Context

The `wikime` skill scaffolds wikis that are well-structured for agents (markdown + frontmatter + lint/query/graph scripts) but offer little to a human reader beyond `index.md` and a force-directed graph (`graph.html`). As we extend the wiki to non-technical end users, the gap is acute: they will not run `python3 scripts/query.py --tag X`, and `index.md` is a bullet list with no filtering, search, or grouping.

A separate brainstorm considered building retrieval/synthesis (semantic search, MCP server, Q&A web app). Each of those introduces operational concerns (hosting, auth, API keys, LLM costs) that would compromise the project's most valuable property today: it is fully agent-agnostic, zero-infrastructure, and works with whatever LLM-coding agent the user has.

Two related directions were filed as deferred backlog items rather than tackled here:
- [PHL-33](https://linear.app/phluxxed/issue/PHL-33/active-autonomous-wiki-maintenance) ŌĆö Active / autonomous wiki maintenance
- [PHL-34](https://linear.app/phluxxed/issue/PHL-34/first-class-ingestion-connectors) ŌĆö First-class ingestion connectors

## Goal

Replace `graph.html` with a single richer artifact, `wiki.html`, that is a complete browseable reader for the wiki. Generated by a script (`scripts/render.py`) that the agent runs after every wiki change, exactly as `scripts/graph.py` is run today. No server, no API, no auth, no LLM calls at render time.

Q&A about the wiki's contents continues to happen in whatever LLM-coding agent the user has (Claude Code, Claude Desktop, Codex, Gemini CLI). The artifact's job is reading and browsing, not answering.

## Non-Goals

These are explicit out-of-scope decisions, recorded so future contributors do not redrift:

- No server, API, or any deployed component.
- No authentication, accounts, or per-user state.
- No semantic / vector search. Full-text only.
- No LLM calls at render time.
- No edit-in-browser. The artifact is read-only.
- No multi-wiki federation, cross-wiki entity reuse.
- No real-time updates. The agent regenerates on demand.
- No mobile-specific UI work. Should remain readable on mobile via responsive defaults; nothing more.

## Delivery channels

The same `wiki.html` file works in three ways with no extra effort:

1. **Local file** ŌĆö `open wiki.html` in any browser. Bookmarkable.
2. **Claude Desktop / claude.ai artifact** ŌĆö drop the HTML into a chat or open as a Claude artifact; renders live.
3. **Static host** ŌĆö push to Netlify, GitHub Pages, S3, or any static target. The team lead deploys once; end users bookmark a URL.

This three-channel property is preserved by keeping the artifact strictly static and self-contained. Any single-file HTML constraint forces this discipline; we should not break it lightly.

## Architecture

One new script, one new artifact, one replaced script.

```
scripts/render.py    (new)      ŌåÆ produces wiki.html
wiki.html            (new)      ŌåÆ single self-contained artifact at wiki root
scripts/graph.py     (removed)  ŌåÆ graph view becomes a tab in wiki.html
graph.html           (removed)  ŌåÆ subsumed
```

`scripts/render.py` data pipeline at render time:

```
collect_pages()       ŌåÆ list of {path, frontmatter, body, rendered_html}
collect_edges()       ŌåÆ wiki-link graph (logic ported from graph.py)
collect_log()         ŌåÆ parse log.md for "Recent changes"
extract_risks()       ŌåÆ all open risks across pages (reuse query.py logic)
extract_open_qs()     ŌåÆ scan pages for `> **Open question:** ...` blockquote convention
build_search_index()  ŌåÆ MiniSearch index, serialised to JSON
render_html()         ŌåÆ single-file HTML with all data, CSS, JS embedded inline
```

### Tech choices

- **Markdown rendering: server-side, in Python** at render time, using the `markdown` library. Rendered HTML is embedded as strings in the data block. Avoids ~40KB of client-side markdown JS, faster page load, simpler. We do not need browser-side markdown manipulation (no edit, no live preview).
- **No JS framework.** Vanilla JS. We are not building an SPA; we are building a static reader with eight tabs.
- **D3** for the graph view. Already in use in `graph.py`; ported across.
- **MiniSearch** (~15KB minified) for client-side full-text search. Index built at render time, queried in-browser. Simpler and more modern than Lunr.js.
- **Inline CSS** continuing the dark theme already established by `graph.html` for visual continuity. No CSS framework.
- **Total artifact size budget: under 500KB** for typical wikis (50-100 pages). Well within Claude artifact limits and reasonable for static hosting.

### Why server-side markdown

Three reasons:
1. **Smaller artifact** ŌĆö no markdown library shipped to the browser.
2. **Faster cold open** ŌĆö no client-side parse step before rendering.
3. **Single source of truth for rendering** ŌĆö the Python `markdown` lib (vs. JS markdown-it) is already a known dependency in the wiki ecosystem; less drift between what the agent generates and what the reader sees.

The cost is a Python dependency at render time (`pip install markdown`). Acceptable: render only runs on the agent's machine, not on end users' machines.

## Views

Eight views in one shell. Left sidebar is a collapsible nav (matches today's panel pattern from `graph.html`). Tag and category filters apply across Home, Search, and Graph.

| View | Source data | Key elements |
| --- | --- | --- |
| **Home** | pages grouped by `category` | Page cards with title, status badge (Draft/Live/Stale), one-line summary, "updated N days ago". |
| **Page** | rendered HTML + frontmatter + edges | Markdown body, frontmatter header (title, status, owner, source, tags, dates), "Mentioned in" sidebar, "Mentions" sidebar, link to source file if any. |
| **Search** | MiniSearch index over title + body + tags | Live filter as you type; result snippets with hit context; click to page view. |
| **Graph** | nodes + edges (from existing graph logic) | Force-directed graph; preserves current filters (orphans toggle, type chips, tag chips, depth slider). |
| **Risks** | extracted open risk rows | Sortable table: page, risk, likelihood, impact, status. Click row ŌåÆ page view. |
| **Recent changes** | parsed `log.md` | Reverse-chronological list, one row per log entry. |
| **Open questions** | scanned `> **Open question:** ...` blockquotes | Page link + question text. Convention is documented in `wiki-agent.md`. |
| **Entities** | pages with `type: entity \| concept` | List with mention-counts; per-entity row links to the entity page and lists pages that mention it. |

### Open question convention

To populate the Open questions view, pages flag unresolved threads with this blockquote pattern:

```markdown
> **Open question:** Does the rate limit reset on a sliding window or fixed window?
```

The render script scans for that exact pattern (`^>\s*\*\*Open question:\*\*\s*(.+)$`). The convention is added to `wiki-agent.md` so the agent uses it consistently when ingesting.

## Integration

### Regeneration model

`wiki-agent.md` adds one rule to the agent's operating manual:

> **After any operation that adds, edits, or removes a wiki page (Ingest, Update, entity creation), run `python3 scripts/render.py`. This is non-optional ŌĆö the artifact must always reflect the current state of the wiki.**

This mirrors the existing pattern (`scripts/lint.py` is run after every ingest today). One additional command in the post-change ritual.

### Documentation updates

- `SKILL.md` ŌĆö directory tree references `wiki.html` and `scripts/render.py`. Scripts table swaps `graph.py` for `render.py`. Adds Open question convention. Adds the regeneration rule to the Operations section. Install step gains `markdown` alongside `pyyaml` (`pip3 install pyyaml markdown`).
- `README.md` (the repo's, not scaffolded wikis') ŌĆö Scripts table updated; install step updated.
- `_templates/CONVENTIONS.md` ŌĆö Scripts table updated. Adds Open question convention to "Keeping the Wiki Healthy".

The scaffolded wiki's `README.md` is generated by inline instructions in `SKILL.md`, so updating `SKILL.md` covers it without needing a separate template file.

## Migration

This is a backward-incompatible change for existing wikis (graph.py / graph.html removed in favour of render.py / wiki.html).

**Existing wiki migration is a one-line action**: replace `scripts/graph.py` with the new `scripts/render.py` from this skill, run it once, delete the now-orphan `graph.html`. We document this in `SKILL.md` and surface it in the next `/wikime` invocation if existing scripts are detected.

**Newly scaffolded wikis** get the new model directly. No transitional code paths.

**Safety net**: `scripts/render.py` produces a valid empty-state artifact when run against a wiki with no pages. Avoids a "first run after scaffold fails" failure mode.

**Updates to existing 3 evals:**

- Replace assertions referencing `graph.html` with `wiki.html`.

**One new eval:**

Out of scope for evals: visual / interactive testing of the artifact in a browser. Defer to manual smoke checks.

## Acceptance signals

- A wiki scaffolded with `/wikime` includes `scripts/render.py` and a `wiki.html` is generated automatically after the first ingest.
- All 8 views work end-to-end on a wiki with at least one page, one entity, and one open risk.
- The agent regenerates `wiki.html` after every ingest without being reminded (because `wiki-agent.md` says so).
- A non-technical end user can open `wiki.html` and find a page by category, search, tag, or click-through, without instructions.
- The artifact opens correctly in Claude Desktop's artifact viewer.
- All 4 evals (3 updated + 1 new) pass cleanly.
